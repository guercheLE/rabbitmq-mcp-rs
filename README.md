# rabbitmq-mcp

RabbitMQ Management HTTP API server for the [Model Context Protocol (MCP)](https://modelcontextprotocol.io/), generated by [mcpify](https://github.com/guercheLE/mcpify).

Instead of placing the complete RabbitMQ API specification in an agent's context, the server exposes three tools:

- `search` finds relevant operations from a natural-language description.
- `get` returns the method, path, documentation, and schemas for one operation.
- `call` validates the arguments, calls RabbitMQ, and validates the response.

The operation catalog and semantic-search vectors are embedded in the binary. The default RabbitMQ 4.3.2 store contains 137 real Management HTTP API operations: 93 `GET`, 20 `DELETE`, 12 `PUT`, and 12 `POST` operations.

## Requirements

- A Rust toolchain compatible with edition 2024
- A reachable RabbitMQ node with the [management plugin](https://www.rabbitmq.com/docs/management) enabled
- A RabbitMQ user allowed to perform the operations you intend to call

## Install

Build the binaries from this checkout:

```bash
cargo build --release
```

The main executable is `target/release/rabbitmq-mcp`. You can also install it on your `PATH`:

```bash
cargo install --path . --bin rabbitmq-mcp
```

## Configure

Run the interactive setup wizard:

```bash
rabbitmq-mcp setup
```

The wizard asks for the RabbitMQ Management HTTP API URL, username and password, RabbitMQ API version, and MCP transport. For a local RabbitMQ installation, the API URL is usually:

```text
http://127.0.0.1:15672
```

Credentials are saved in the operating-system keychain when available. If no keychain service is available, the server uses an encrypted file under `~/.rabbitmq-mcp/`. The wizard also prints an MCP client configuration and can write the non-secret connection settings to `.env`.

When using the generated `.env` with the terminal client, load it into the current shell first:

```bash
set -a
source .env
set +a
```

Verify the resulting connection and inspect the active non-secret configuration:

```bash
rabbitmq-mcp test-connection
rabbitmq-mcp config
rabbitmq-mcp versions
```

The supported API versions are `4.3.2` (default), `4.2.8`, `4.1.8`, `4.0.9`, and `3.13.7`. Set `RABBITMQ_MCP_API_VERSION` to select a non-default store.

## Try the terminal client

The same executable provides direct `search`, `get`, and `call` commands. This is useful for exploring the catalog and checking an operation before exposing the server to an MCP client.

### 1. Find an operation

```bash
rabbitmq-mcp search "list queues" --limit 3
```

Using the default `mcp_store.db`, this returns:

```json
[
  {
    "operation_id": "getApiQueuesDetailed",
    "similarity": 0.0988585352897644,
    "summary": "A list of all queues containing all available information about the queues (over 50 fields per queue)."
  },
  {
    "operation_id": "getApiQueuesVhost",
    "similarity": 0.0870896577835083,
    "summary": "A list of all queues in the given virtual host containing all available information about the queues (over 50 fields per queue).."
  },
  {
    "operation_id": "getApiQueues",
    "similarity": 0.04965442419052124,
    "summary": "A list of all queues across all virtual hosts returning a reduced set of fields."
  }
]
```

Similarity values are ranking aids rather than probabilities.

### 2. Inspect its contract

Use the exact `operation_id` returned by `search`:

```bash
rabbitmq-mcp get getApiQueuesVhost
```

`get` prints the stored HTTP method and path, input parameters, response schema, documentation, and authentication scheme. Inspecting an operation before calling it is especially useful for request bodies and version-specific fields.

For example, the catalog describes `putApiQueuesVhostName` as:

```json
{
  "operation_id": "putApiQueuesVhostName",
  "method": "PUT",
  "path": "/api/queues/{vhost}/{name}",
  "summary": "An individual queue.",
  "auth_scheme_ref": "basicAuth"
}
```

Its full `get` result also includes the path parameters and the observed request-body example.

### 3. Call RabbitMQ

Pass one JSON object through `--args` (or `-a`). Path, query, and header parameters are top-level properties. Put a JSON request body under `body`.

Read the overview:

```bash
rabbitmq-mcp call getApiOverview
```

List queues with real query parameters from the default store:

```bash
rabbitmq-mcp call getApiQueues --args '{
  "page": "1",
  "page_size": "20",
  "pagination": "true"
}'
```

Read one queue. The default virtual host is `/`; the server percent-encodes it when inserting it into the URL:

```bash
rabbitmq-mcp call getApiQueuesVhostName --args '{
  "vhost": "/",
  "name": "demo.queue"
}'
```

Create or update a durable queue:

```bash
rabbitmq-mcp call putApiQueuesVhostName --args '{
  "vhost": "/",
  "name": "demo.queue",
  "body": {
    "durable": true,
    "auto_delete": false,
    "arguments": {}
  }
}'
```

Retrieve up to five messages while putting them back on the queue:

```bash
rabbitmq-mcp call postApiQueuesVhostNameGet --args '{
  "vhost": "/",
  "name": "demo.queue",
  "body": {
    "count": 5,
    "ackmode": "ack_requeue_true",
    "encoding": "auto",
    "truncate": 50000
  }
}'
```

`call` performs a live request and can change RabbitMQ state. Review `PUT`, `POST`, and `DELETE` operations with `get` before running them.

## Connect an MCP client

### stdio

After running `rabbitmq-mcp setup`, configure an MCP host to spawn the server. Include the connection settings printed by the wizard; this example targets a local RabbitMQ 4.3.2 node:

```json
{
  "mcpServers": {
    "rabbitmq": {
      "command": "rabbitmq-mcp",
      "args": ["start"],
      "env": {
        "RABBITMQ_MCP_URL": "http://127.0.0.1:15672",
        "RABBITMQ_MCP_AUTH_METHOD": "basic",
        "RABBITMQ_MCP_API_VERSION": "4.3.2",
        "RABBITMQ_MCP_TRANSPORT": "stdio"
      }
    }
  }
}
```

Use the absolute executable path if `rabbitmq-mcp` is not on the MCP host's `PATH`. The stdio server reads the connection settings from this `env` block and uses the credentials saved by `setup`.

An effective agent workflow is:

1. Call `search` with the desired task, such as `find queues with messages`.
2. Call `get` with a returned `operation_id`.
3. Construct arguments from the returned schema and call `call`.

### HTTP

Start a standalone Streamable HTTP server:

```bash
rabbitmq-mcp http --host 127.0.0.1 --port 3000
```

The MCP endpoint is `http://127.0.0.1:3000/mcp`. Every HTTP request must include the RabbitMQ Basic Authentication value; HTTP transport intentionally does not fall back to credentials stored on the server:

```json
{
  "mcpServers": {
    "rabbitmq": {
      "url": "http://127.0.0.1:3000/mcp",
      "headers": {
        "Authorization": "Basic <base64-encoded-username-and-password>"
      }
    }
  }
}
```

Keep the listener on localhost unless you have added appropriate network access controls and TLS in front of it.

## Docker

Create `.env` from the provided example, fill in the connection settings, and build the image:

```bash
cp .env.example .env
docker compose build
```

Start either the stdio-oriented service or the HTTP service:

```bash
docker compose run --rm rabbitmq-mcp start
docker compose up rabbitmq-mcp-http
```

The HTTP service listens on port 3000. The Compose file mounts `~/.rabbitmq-mcp` into the container so credentials produced by setup can persist.

## Embedded API data

The default `mcp_store.db` targets RabbitMQ 4.3.2. It contains the operation identifiers, HTTP methods and paths, descriptions, input/output schemas, authentication references, and 768-dimensional semantic-search vectors used by `search`.

Versioned stores provide the same role for older RabbitMQ releases:

| RabbitMQ API version | Embedded store |
| --- | --- |
| 4.3.2 (default) | `mcp_store.db` |
| 4.2.8 | `mcp_store_v4.2.8.db` |
| 4.1.8 | `mcp_store_v4.1.8.db` |
| 4.0.9 | `mcp_store_v4.0.9.db` |
| 3.13.7 | `mcp_store_v3.13.7.db` |

The database is an embedded runtime artifact; users do not need to install SQLite or place a `.db` file beside the installed executable.

The corresponding OpenAPI documents and their provenance are available in [`openapi/`](openapi/) and [`docs/api-sources.md`](docs/api-sources.md).

## Development

Run the test suite:

```bash
cargo test
```

Generate an HTML coverage report (requires `cargo-llvm-cov`):

```bash
bash scripts/coverage.sh
```

The report is written to `target/coverage/html/index.html`.

Collect a CPU profile with `samply` and write `profile/bottleneck-report.md`:

```bash
cargo install samply
bash scripts/profile.sh
```

Collect a heap profile with the optional `profiling` feature:

```bash
cargo run --release --features profiling -- search "list queues"
```

The heap report is written to `dhat-heap.json`.
